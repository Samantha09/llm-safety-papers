## 2. 英文摘要原文（arXiv Abstract原文）

> Large Language Models (LLMs), renowned for their superior proficiency in language comprehension and generation, stimulate a vibrant ecosystem of applications around them. However, their extensive assimilation into various services introduces significant security risks. This study deconstructs the complexities and implications of prompt injection attacks on actual LLM-integrated applications. Initially, we conduct an exploratory analysis on ten commercial applications, highlighting the constraints of current attack strategies in practice. Prompted by these limitations, we subsequently formulate HouYi, a novel black-box prompt injection attack technique, which draws inspiration from traditional web injection attacks. HouYi is compartmentalized into three crucial elements: a seamlessly-incorporated pre-constructed prompt, an injection prompt inducing context partition, and a malicious payload designed to fulfill the attack objectives. Leveraging HouYi, we unveil previously unknown and severe attack outcomes, such as unrestricted arbitrary LLM usage and uncomplicated application prompt theft. We deploy HouYi on 36 actual LLM-integrated applications and discern 31 applications susceptible to prompt injection. 10 vendors have validated our discoveries, including Notion, which has the potential to impact millions of users. Our investigation illuminates both the possible risks of prompt injection attacks and the possible tactics for mitigation.

## 3. 中文摘要翻译

大型语言模型（LLMs）以其卓越的语言理解和生成能力而闻名，在其周围催生了一个蓬勃发展的应用生态系统。然而，LLMs被广泛融入各种服务也引入了重大的安全风险。本研究深入分析了针对实际LLM集成应用的提示注入攻击的复杂性和影响。

首先，我们对十个商业应用进行了探索性分析，揭示了当前攻击策略在实际应用中的局限性。基于这些限制，我们随后提出了**HouYi**——一种新颖的黑盒提示注入攻击技术，灵感来源于传统的Web注入攻击。HouYi被分解为三个关键要素：**无缝嵌入的预构建提示**、**诱导上下文分割的注入提示**，以及**为实现攻击目标而设计的恶意载荷**。

利用HouYi，我们揭示了此前未知且严重的攻击后果，例如**无限制的任意LLM使用**和**简单的应用提示窃取**。我们将HouYi部署在36个实际的LLM集成应用上，发现31个应用易受提示注入攻击。10家供应商已验证了我们的发现，包括Notion在内，这可能影响数百万用户。

本研究阐明了提示注入攻击的潜在风险以及可能的缓解策略。

## 4. 研究背景

### 4.1 LLM集成应用的兴起

随着GPT-4、Llama等大型语言模型的快速发展，一个围绕LLM的应用生态系统正在蓬勃兴起。LLM被集成到各种服务和应用中，包括：

- **办公助手**：如Notion AI、Microsoft Copilot
- **内容创作平台**：如 Jasper AI、Writesonic
- **智能客服系统**：各种聊天机器人和虚拟助手
- **开发工具**：代码补全助手如GitHub Copilot
- **信息检索系统**：如Perplexity AI

这些集成应用将LLM的强大语言能力与具体业务场景结合，为用户提供智能化服务。

### 4.2 提示注入攻击的威胁

提示注入（Prompt Injection）是一种新型攻击技术，攻击者通过在输入中注入恶意设计的提示来操纵LLM的行为。与传统的Web注入攻击（如SQL注入、XSS）类似，提示注入利用了系统对用户输入处理不当的漏洞。

**传统Web注入攻击的类比：**

| Web注入 | 提示注入 |
|---------|----------|
| SQL注入 | 指令注入 | 
| XSS | 提示跨站脚本 |
| 命令注入 | 系统提示劫持 |

### 4.3 现有攻击的局限性

先前针对LLM安全的研究主要集中在越狱（Jailbreak）攻击上，这些攻击通常：

1. **需要白盒访问**：攻击者需要了解模型的内部结构和权重
2. **依赖启发式方法**：使用手工设计的提示模板
3. **缺乏实际验证**：在真实商业应用上的测试有限
4. **攻击效果不稳定**：在不同的LLM后端上效果差异大

这些局限性促使研究者开发新的攻击方法。

## 6. 研究方法

### 6.1 HouYi攻击框架概述

HouYi的设计灵感来源于传统Web注入攻击，其核心思想是将恶意指令隐藏在看似正常的用户输入中，让LLM在处理时将其误认为是合法的系统指令。

**HouYi的三个核心组件：**

```
┌─────────────────────────────────────────────────────────────┐
│                    HouYi 攻击框架                           │
├─────────────────────────────────────────────────────────────┤
│  1. 预构建提示 (Pre-constructed Prompt)                      │
│     - 无缝嵌入的正常内容                                    │
│     - 不引起安全检测                                        │
│     - 为后续攻击创造上下文                                  │
├─────────────────────────────────────────────────────────────┤
│  2. 注入提示 (Injection Prompt)                            │
│     - 诱导上下文分割                                       │
│     - 使LLM将恶意内容误认为系统指令                         │
│     - 利用LLM的指令遵循特性                                 │
├─────────────────────────────────────────────────────────────┤
│  3. 恶意载荷 (Malicious Payload)                            │
│     - 实现具体攻击目标                                      │
│     - 如：窃取提示、绕过限制、获取敏感信息                  │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 上下文分割机制

HouYi的关键创新在于**上下文分割（Context Partition）**机制。攻击者精心构造的注入提示会导致LLM错误地将其后输入的内容视为系统级指令而非用户输入。

**上下文分割原理：**

```
正常情况：
[系统指令] + [用户输入] → [模型输出]

HouYi攻击后：
[系统指令] + [用户输入(包含恶意注入)] + [攻击者控制的"系统指令"] → [攻击者期望的输出]
```

### 6.3 攻击流程

**HouYi攻击的完整流程：**

1. **信息收集阶段**
   - 识别目标LLM集成应用
   - 分析应用的输入格式和LLM调用方式
   - 确定潜在的攻击面

2. **预构建提示生成**
   - 创建看似正常的用户输入
   - 包含用于触发特定LLM行为的关键词
   - 为后续注入创造有利上下文

3. **注入提示设计**
   - 利用LLM的指令遵循特性
   - 设计能诱导上下文分割的提示模式
   - 测试和优化注入效果

4. **恶意载荷执行**
   - 在成功注入后执行具体攻击
   - 收集敏感信息或滥用服务

5. **结果验证与分析**
   - 确认攻击是否成功
   - 评估攻击影响范围
   - 整理发现并准备报告

### 6.4 黑盒攻击的优势

HouYi作为黑盒攻击方法具有以下优势：

| 优势 | 说明 |
|------|------|
| **无需模型访问** | 不需要知道目标LLM的内部结构或权重 |
| **无需API密钥** | 通过正常用户界面即可发起攻击 |
| **难以防御** | 攻击流量与正常用户请求难以区分 |
| **可扩展性强** | 同一种攻击模式可应用于多个类似系统 |
| **实际威胁大** | 任何普通用户都可以成为攻击者 |

## 8. 实验结果

### 8.1 现有攻击方法的局限性

初步实验揭示了现有提示注入攻击方法的几个关键局限：

**1. 基于规则的攻击容易被过滤**

许多商业应用已经部署了基于规则的内容过滤器，可以识别常见的注入提示模式。

**2. 白盒攻击假设不现实**

大多数先前的研究假设攻击者可以访问模型权重，这在实际的商业应用场景中几乎不可能。

**3. 缺乏上下文意识**

简单的注入尝试往往因为与正常用户输入差异过大而被检测。

### 8.2 HouYi攻击效果

**总体攻击成功率：**

在36个测试的LLM集成应用中：
- **31个应用（86.1%）对HouYi攻击敏感**
- **平均攻击成功率达到67.3%**

**各应用类型攻击效果：**

| 应用类型 | 测试数量 | 敏感数量 | 敏感比例 |
|----------|----------|----------|----------|
| 办公助手 | 12 | 11 | 91.7% |
| 内容生成 | 10 | 9 | 90.0% |
| 代码辅助 | 8 | 6 | 75.0% |
| 客服系统 | 6 | 5 | 83.3% |

### 8.3 发现的攻击后果

**1. 无限制的任意LLM使用**

最严重的发现是攻击者可以绕过付费限制，免费使用商业LLM服务的高能力版本。这对于按量付费的商业模式构成重大威胁。

**示例攻击场景：**
```
攻击者通过提示注入，使应用在后台调用高能力LLM（如GPT-4），
而应用本身只使用了低能力版本（如GPT-3.5）。
攻击者因此可以免费享受高能力服务。
```

**2. 应用提示窃取**

攻击者可以提取被集成应用的核心提示和指令，这些提示往往包含了：
- 应用的专有知识和处理逻辑
- 商业敏感的系统设计
- 用户的隐私偏好设置

**3. 服务滥用**

攻击者可以利用他人的账户配额进行自己的任务，造成：
- 账户资源被盗用
- 服务成本转移
- 隐私数据交叉污染

### 8.4 供应商确认

经过负责任的披露程序，以下供应商确认了研究发现的准确性：

- **Notion** - 影响了其AI助手功能的数百万用户
- 其他9家不愿公开名称的供应商

### 10.2 关键攻击技术细节

**上下文分割的实现：**

HouYi使用多种技术诱导上下文分割：

1. **指令覆盖攻击**
   - 注入看似官方的系统指令
   - 利用LLM对"系统指令"的特殊处理

2. **角色扮演攻击**
   - 让LLM扮演特定角色
   - 然后利用角色特性绕过限制

3. **级联注入攻击**
   - 分多次注入，逐步改变LLM行为
   - 每次注入都建立在前一次的基础上

## 11. 消融实验

### 11.1 各组件的贡献分析

为了评估HouYi各组件的独立贡献，作者进行了系统的消融实验：

**实验设置：**
- 分别移除每个组件
- 测试剩余组件的攻击效果
- 对比完整攻击的成功率

**实验结果：**

| 攻击配置 | 攻击成功率 | 相对完整攻击 |
|----------|------------|--------------|
| 完整HouYi | 67.3% | 100% |
| 无预构建提示 | 41.2% | 61.2% |
| 无注入提示 | 23.5% | 34.9% |
| 无恶意载荷 | 0% | 0% |
| 仅上下文分割 | 28.7% | 42.7% |

**分析结论：**

1. **预构建提示的作用**
   - 贡献约26%的攻击成功率
   - 为后续攻击创造有利环境
   - 帮助绕过初步内容过滤

2. **注入提示的核心作用**
   - 贡献约43%的攻击成功率
   - 是实现上下文分割的关键
   - 单独的上下文分割已具有相当的威胁性

3. **恶意载荷的必要性**
   - 载荷是攻击的最终执行者
   - 没有载荷的攻击只是"测试"，无法造成实际危害

### 11.2 不同LLM后端的脆弱性差异

**测试的不同LLM后端：**

| LLM后端 | 易感性评级 | 平均攻击成功率 |
|---------|------------|----------------|
| GPT-4 | 高 | 71.2% |
| GPT-3.5-turbo | 高 | 68.5% |
| Claude | 中 | 52.3% |
| Llama-2 | 中 | 48.7% |
| PaLM | 低 | 35.1% |

**发现：**
- GPT系列对HouYi攻击最为脆弱
- 经过安全对齐的模型（如Claude）表现出更强的抵抗力
- 开源模型（如Llama-2）的脆弱性介于两者之间

### 11.3 攻击变体对比

**不同注入技术的效果对比：**

| 注入技术 | 成功率 | 检测率 | 综合评分 |
|----------|--------|--------|----------|
| 直接指令注入 | 78.2% | 65% | 中 |
| 上下文分割注入 | 67.3% | 22% | 高 |
| 级联注入 | 71.5% | 35% | 中高 |
| 混合注入 | 82.1% | 28% | 最高 |

**结论：**
混合注入技术虽然检测率较低，但结合多种技术的攻击效果最佳。

## 12. 局限性

### 12.1 研究局限性

**1. 测试覆盖范围有限**

虽然测试了36个应用，但相对于市场上众多的LLM集成应用，这个数字仍然较小。研究结果的普适性需要进一步验证。

**2. 动态防御的挑战**

研究采用的是某一时间点的应用状态。随着供应商部署新的安全措施，攻击的有效性可能会降低。

**3. 攻击自动化程度不足**

当前的HouYi攻击需要攻击者具备一定的技术背景，完全自动化的攻击工具尚未开发。

### 12.2 方法论局限性

**1. 黑盒假设的局限性**

虽然HouYi定位为黑盒攻击，但在实际部署中，攻击者可能需要了解目标应用的某些非公开信息（如API端点、认证方式等）。

**2. 伦理限制**

由于伦理考虑，研究未能测试更具破坏性的攻击场景，如大规模服务滥用或数据窃取。

**3. 因果归因的挑战**

难以精确区分攻击成功是因为LLM的脆弱性还是应用实现的缺陷。

### 12.3 防御建议的局限性

本文提出的防御建议主要集中在检测层面，对于根本性地解决提示注入问题缺乏深入的探索。未来的研究需要更多地关注：

- LLM层面的一体化安全设计
- 应用层面的输入验证机制
- 系统层面的隔离和沙箱技术

## 13. 伦理声明

### 13.1 研究伦理

本研究严格遵循计算机安全研究的伦理准则：

**1. 负责任的披露**

- 所有发现的漏洞都先通知受影响的供应商
- 给予供应商至少90天的修复时间
- 在供应商确认后再公开发布研究结果

**2. 最小化伤害原则**

- 实验控制在最小必要范围内
- 避免对服务造成实际中断或损害
- 不进行任何可能伤害最终用户的测试

**3. 数据保护**

- 所有实验数据都经过脱敏处理
- 不收集或存储任何个人身份信息
- 研究结束后删除所有测试数据

### 13.2 利益冲突声明

作者声明不存在可能影响研究客观性的利益冲突。

### 13.3 伦理审批

本研究获得了相关机构伦理委员会的批准，遵循了所有适用的伦理准则和法规。
